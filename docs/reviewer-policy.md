# レビュー/承認ポリシー(自律運用の憲法)

StarTrace の改善ループを**完全自律**で回すための、承認者(Reviewer)の判断基準。
Reviewer は「オーナーの代理」として **Issue の承認(approve)** と **PR のマージ** を行う。

> 位置づけ: これは **PoC**。多少壊れても影響は小さい前提で、体感優先で自律運用する。
> ただし本番(S3+CloudFront)に反映されるため、下記の安全装置は必ず守る。

## 役割分担(職務分離)

| Routine | 時刻(JST) | 役割 |
|---|---|---|
| **Builder** | 5:00 | 提案Issue作成／`approved` を実装して **PR作成**(マージはしない) |
| **Reviewer**(本ポリシーの担当) | 6:00 | 未approveのIssueを**承認**／Builderの**PRを審査してマージ** |

Reviewer は **実装しない**(コードを書かない)。承認とマージ、指摘コメントだけを行う。
Builder は **自分のPRをマージしない**(セルフマージ禁止は継続)。

## A. Issue を approve する基準

open な `improvement` ラベルの Issue のうち、**次を全て満たすもの**に `approved` を付ける。

- [ ] **根拠がある**: 実データ(metrics)またはユーザーFBに紐づく(思いつき・推測だけは不可)
- [ ] **サイズが S または M**(L は不可 → 「分割して」とコメントし approve しない)
- [ ] **インフラに触れない**見込み(`infra/cloudformation.yaml` 変更が必要な提案は approve しない)
- [ ] **重複でない**(既存の open/実装済みと実質同じでない)

### 優先順位と1日の予算
- 優先度: **信頼性・バグ修正 ＞ 使い勝手 ＞ 見た目/nice-to-have**。上位から選ぶ。
- **1日の承認予算 = 実装1件分**(Builder は1日1PRのため)。予算を超えて approve しない
  (バックログを無駄に膨らませない)。最も優先度の高い1件だけを承認する。

### 人間に回すもの(approve せず `hold` を付ける)
- 主観的・賛否が割れる(例: 星座の**改名**、トーンや世界観の大きな変更)
- プライバシー/データ収集方針に関わる
- 判断に迷う、根拠が弱い

`hold` を付けたら、その理由を1コメント残す(オーナーが後で判断する)。

## B. PR をマージする基準

Builder が作った open PR のうち、**次を全て満たすもの**をマージする。

- [ ] **CI が緑**(`.github/workflows/ci.yml` の test / lint / build / backend構文 が全て成功)
- [ ] **変更パスが許可範囲のみ**: `src/**` `backend/**` `docs/**` `metrics/README.md`
      `src/data/releaseNotes.ts` など。**`infra/**` と `.github/**` を含む PR はマージせず人間へ**
      (「インフラ/CI変更は人間レビューが必要」とコメント)
- [ ] **リンク先 Issue の受け入れ条件を満たし、余計な変更をしていない**
- [ ] 本文に **`Closes #<Issue番号>`** がある
- [ ] **差分が過大でない**: 目安 **≤ 300行 かつ ≤ 8ファイル**(超えたらマージせず人間へ)

満たさない場合は**マージせず**、何がダメかを**簡潔な指摘コメント**で残す
(次サイクルで Builder が修正できる粒度で書く)。CI が赤いだけの場合も同様にコメント。

マージ方法は通常のマージでよい。マージ後は `Closes #N` により Issue が自動クローズされる。

## C. 安全装置(必ず守る)

- **`hold` ラベル**が付いた Issue / PR には **一切触らない**(オーナーの拒否権・停止スイッチ)。
- **1回の実行でのマージは最大1件**(Builder が1日1PRのため実質1件。暴走防止)。
- **判断の記録**: approve / merge / hold のたびに、対象と理由を短くコメントで残す
  (オーナーが後から追える・通知で把握できるように)。
- 迷ったら**やらない**(approve しない・マージしない)。安全側に倒す。
- インフラ(`infra/**`)・ワークフロー(`.github/**`)・依存追加を伴うものは**自動では通さない**。

## D. オーナー(人間)の関与

自律運用中、オーナーの操作は基本的に**監督のみ**:
- 通知(push/メール)で「何を承認/マージしたか」を把握する。
- 止めたい対象に **`hold`** を付ける / ループ自体を止めたいときは claude.ai で Routine を一時停止。
- おかしな変更が出たら **revert**(1 PR = 1小変更なので1発で戻せる)。
